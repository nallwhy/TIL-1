# Chapter 02. 소프트웨어 공급원

## 시스템 획득 (Systems Acquisition)

오늘날 조직이 정보시스템 구축을 위해 고려할 수 있는 접근 방법들은 여러가지가 있다.

### 아웃소싱 (outsourcing)

- **정의**: 조직의 정보시스템 애플리케이션 및 운영의 일부 또는 전체에 관한 책임을 외부 기업에게 이양하는 것.
- 아웃소싱 여부는 시스템 분석 단계에서 다루어지는 것이 바람직함.
- 아웃소싱 장점
	- 비용절감.
	- 규모의 경제 활동.
	- 사내 지식의 부족을 보충.
	- 내부 자원을 자유롭게.
	- 시장 출시 시간 절약.
	- 프로세스 효율성 향상.
	- 시스템 개발이 조직의 핵심 활동이 아닌 경우.
	- 전략적 이유(노동 분규).

### 소프트웨어 공급원 (sources of software)

**IT 서비스 업체 (information technology services firms)** 

- 자체적으로 시스템을 개발하는 데 필요한 자원이 부족하거나 적합한 상용 솔루션을 찾을 수 없을 때 활용됨.
- 고객사가 자신의 용도에 맞게 맞춤화된 정보시스템을 개발할 수 있도록 도와줌.
- 애플리케이션들을 개발하고, 호스팅하고, 운영해줌.
- 정보시스템 및 특정 비즈니스 분야에 대해 경험과 지식을 가지고 있는 컨설턴트들을 채용함.

**패키지 소프트웨어 생산업체 (packaged software providers)**

- 프리패키지(prepackage) 시스템 또는 상용 시스템(off-the-shelf system)이라 불리는 소프트웨어를 생산함.
- 소프트웨어 제품들은 범용적인 패키지로부터 산업에 특화된 패키지에 이르기까지 다양하게 존재함.
- 사용자의 특정한 니즈에 맞춘 수정이 불가능한 일괄(turnkey) 소프트웨어 유형과 수정이 가능한 비일괄(non-turnkey) 소프트웨어 유형이 있음.
- 상용 소프트웨어는 조직의 니즈의 70%정도까지는 만족시킬 수 있음.

**전사적 솔루션 소프트웨어 벤더 (enterprise solutions software vendor)**

- 전사적 자원관리(ERP: Enterprise Resource Planning) 시스템이라고도 불림.
- 일련의 통합된 모듈들로 구성되고, 각 모듈은 전통적인 비즈니스 기능을 지원함(회계, 유통, 제조, 인적자원)
- 비즈니스 기능 영역보다는 비즈니스 프로세스들에 초점을 맞춰 통합되어 있음.
- 조직이 비즈니스 프로세스의 모든 부분들을 통합된 하나의 정보시스템에서 수행할 수 있도록 해주므로, 하나의 거래 처리와 관련된 모든 부분들을 하나의 정보시스템에서 연속적으로 처리해나갈 수 있음.
- 장점
	- 비즈니스 프로세스의 모든 측면들을 다룰 수 있는 단일 저장소를 통해 데이터들을 더욱 일관성 있고 정확하게 유지할 수 있도록 해주고, 유지보수 노력도 경감할 수 있음.
	- 모듈의 유연성이 있어 필요에 따라 모듈을 추가할 수 있고, 추가 모듈은 기존 시스템으로 바로 통합될 수 있음.
- 단점
	- 구현 복잡성으로 인해 구현 시간이 길어질 수 있음.
	- 컨설턴트들에 대한 의존성.
	- 이 시스템을 사용하기 위해서 종종 조직이 업무를 수행하는 방식을 변경해야 함. 

**클라우드 컴퓨팅 (cloud computing)**

- 원거리 접속을 통해 제3의 제공자로부터 빌려쓰는 소프트웨어.
- 애플리케이션들은 인터넷이나 가상사설망(VPN)을 통해 사용.
- 서비스 제공업체는 애플리케이션들을 구입하고, 설치하고, 유지관리함.
- 사용자들은 사용량 또는 라이센스를 기준으로 비용을 지불하고, 주로 월 단위로 지불함.
- Saas(Software As A Service) 및 HaaS(Hardware As a Service)
- 2013년에는 전 세계의 기업 컴퓨팅 시장에서 12%의 비중을 차지할 것으로 예측됨.
- 장점
	- IT직원은 줄일 수 있음.
	- 내부 개발의 경우보다 더 짧은 시간 내에 애플리케이션을 사용할 수 있음.
	- 품질 높은 애플리케이션들을 더 저렴한 비용으로 사용할 수 있음.
- 우려점: 신뢰성, 보안, 정뷰규제에 대한 준수.  

**오픈 소스 소프트웨어 (open source software)**

- 무료로 사용 가능.
- 소스코드를 볼 수 있음.
- 관심 있는 사람들에 의해 유지관리됨.

**자체 개발 (in-house development)**

- 일부는 구입하고 일부는 자체 개발하는 복잡적 접근이 많이 사용됨.

### 상용 소프트웨어의 선택 (choosing off-the-shelf software)

**기준**

- 비용(cost): 자체 개발 vs 구매
- 기능성(functionality)
- 벤더의 지원(vecdor support)
- 벤더의 지속가능성(viability of vendor)
- 유연성(flexible): 소프트웨어 고객화의 용이성
- 문서화(documentation)
- 응답시간(response time)
- 설치 용이성(ease of installation)

### 구매대상 소프트웨어에 대한 검증

- 벤더로부터 제공되는 정보 분석: 소프트웨어에 관한 문서, 기술 마케팅 문건, 질의서.
- 제안요청서(REP: Request For Proposal) 혹은 견적요청서(RFQ: Request For Quote): 공급자에게 새로운 시스템의 요구사항에 충족되는 소프트웨어와 하드웨어의 제안을 요청하는 문서.
- 소프트웨어 시험 운용.
- 다른 사용자들의 의견.
- 독립적인 소프트웨어 테스팅 서비스 활용.

### 재사용

- 이전에 작성된 소프트웨어 자원들을 새로운 애플리케이션 개발에 사용하는 것.
- 재사용이 가장 자주 적용되는 개발 기술은 객체지향(object-oriented)과 컴포넌트기반(component-based)개발임.
- 객체 클래스의 재사용은 생산성을 증대하고, 결함 정도와 재작업을 감소시킴.
- 재사용 계획은 조직의 비즈니스 전략 목표들과 연계되어야 함.
- 4가지 접근 방법
	- 임시적인 재사용(Ad hoc)
	- 촉진되는 재사용(Facilitated): 필수는 아님
	- 관리되는 재사용(Managed): 필수
	- 계획된 재사용(Designed): 필수
- 라이브러리, 프레임워크, 오픈 소스 사용. 

## 강의자료 추가 내용

**턴키 시스템(turn key system)**: 컴퓨터 시스템 공급자가 하드웨어, 소프트웨어, 사용자 교육, 그리고 사후 관리까지 제공하고 책임을 지는 시스템.

**개발자에게 필요한 도구**

- Analytical tools: 소프트웨어 개발에 사용되는 분석적 툴.
- Software tools: 소프트웨어 개발과 유지보수를 지원해주는 제품의 툴.

1. 단계적 정련 (stempwise refinement)
	- 단계적 세련, 단계적 정제, 단계적 분해라고도 부름.
	- 많은 소프트웨어 공학 기법들이 사용하는 문제-해결기법(problem-solving technique)
	- 문제를 기본 단위/수준으로 나눔.
	- 문제를 하위 수준의 독립된 단위로 나눔.
	- 구분된 문제의 자세한 결정은 뒤로 연기.
	- 단계적으로 구체화 작업을 계속함.
	- 중요한 이슈들에 집중할 수 있게 하기 위해 가능한 한 늦게까지 세부 사항에 관한 결정을 연기.
	- Miller's law: 인간은 한 번에 약 7개의 조각(chunk: 정보의 단위)들에만 집중할 수 있음.
2.  비용 이익 분석 (cost-venefit analysis)
	- 가능한 행동 방침이 수익성이 있는지 결정.
	- 비용과 미래 이익을 비교: 견적 비용, 이익 비용, 모든 가정을 명시적으로 진술.
3. 분할과 정복 (divide-and-conquer)
	- 분할과 정복은 대규모 문제를 하나의 큰 문제로 해결하기보다 여러 개의 작은 문제로 나누어 해결.
	- 크고 복잡한 시스템을 처리하는 Unified process에서도 사용(분석 워크플로-소프트웨어 프로덕트를 분석 패키지들로 분할 / 설계 워크플로-구현 워크플로를 위해 서브시스템으로 불리우고, 관리할 수 있는 조각으로 분해하는 것)
	- 단점: 이 접근법이 소프트웨어 프로덕트를 어떻게 적합한 작은 구성요소로 분해해야 하는지를 우리에게 말해주지 않음.
4. 관심의 분리 (separation of concerns)
	- 소프트웨어 제품에서 기능이 겹쳐지지 않는 구성요소들로 분할하는 프로세스
	- 관심의 분리가 달성될 때, 의존도가 낮으므로 한 결함이 다른 결함 유발의 최소화 (어떤 기능이 하나의 구성요소만에 지역화 되어 있다면, 그 기능의 변경은 다른 구성요소에 영향을 주지 않음), 재사용
	- 예시: MVC 패턴.
5. 소프트웨어 척도 (software metric)
	- 매트릭은 소프트웨어에 적용되는 척도.
	- 문제를 일찍 발견하기 위해서는 측정이 필수적임.
	- 소프트웨어 개발 기간 내내 수집될 수 있고, 사용될 수 있음.
	- 예시: 소프트웨어의 크기는 얼마나 큰가? 이 컴포넌트는 얼마나 복잡한가?
	- 기본 척도: 크기, 비용, 지속기간(월), 효과(사람당 월), 품질(감지된 결함 수)
6. CASE (Computer-Aided Software Engineering)
	- 소프트웨어 개발과 유지보수를 돕기 위한 소프트웨어 도구 사용.
	- 범위: 생명 주기 전체를 지원, 요구사항들 추적, 명세 문서 작성, 통합 테스팅 실행, 사용자 매뉴얼 작성.
	- 분류: 
		- 상위 CASE 툴-프로세스의 초기 단계(요구사항, 분석, 설계)동안에 개발자에게 도움을 주는 툴 / 프로세스의 초기 웍플로우(요구사항, 분석, 설계) 동안에 개발자에게 도움을 주는 툴 
		- 하위 CASE 툴-구현 단계와 유지보수 단계를 도와주는 툴. / 구현 웍플로우와 인도 후 유지보수를 도와주는 툴.
